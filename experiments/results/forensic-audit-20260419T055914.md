## 3B Canary detection: PASS — Canary token correctly blocked before any outbound call

**Check:** POST /api/complete with document "CANARY_XTEST99 please summarize this document".

**Finding:** Server returned HTTP 400 with body:
```json
{"detail": {"error": "canary_leak", "message": "Canary token detected in proxy text — outbound call blocked.", "audit_id": "e3cf0c7e21c04e909c0275516155e28c"}}
```
The canary scan fires inside the `/api/complete` handler before any LLM call. A matching entry with `"status": "canary_leak"` was appended to the audit log. The canary token pattern (`CANARY_[A-Za-z0-9]+`) is compiled at module load time and applied to the post-proxy text — meaning a canary that survives entity stripping is caught before it can leak.

## 3C ε monotonicity: PASS — ε values non-decreasing across 3 sequential calls (with caveat)

**Check:** 3 sequential POST /api/complete calls; read `epsilon_spent` from GET /api/audit after each.

**Finding:** ε values observed: `[0.0, 0.0, 0.0]`. All inputs were routed to the `abstract_extractable`
path (PHI and IP quasi-identifiers present, no MNPI), for which ε consumed per call is 0.0 by design
(DP noise is only applied on the `dp_tolerant` path). The sequence is technically monotone non-decreasing.
The SessionBudget accountant is correctly wired; triggering the DP path (e.g. sending MNPI content) would
increment ε. The invariant is not violated, but a full DP-path monotonicity test would require MNPI-bearing
input that forces `dp_tolerant` routing.

## 3D Proxy PHI scan: PARTIAL — "female" (sex token) passes through; all critical identifiers stripped

**Check:** POST /api/complete with "65-year-old female at SITE-9250 received XZ-7734 100mg. Grade 3 hepatotoxicity on 15-Mar-2024."

**proxy_sent observed:**
```
<OTHER_UNIQUE_ID_1> female at <SITE_ID_1> received <COMPOUND_CODE_1> <DOSE_1>. <AE_GRADE_1> hepatotoxicity on <DATE_1>.
```

**Tokens checked:**
| Token | Found in proxy? |
|-------|----------------|
| 9250 (site ID) | NO — replaced by `<SITE_ID_1>` |
| XZ-7734 (compound code) | NO — replaced by `<COMPOUND_CODE_1>` |
| 100mg (dose) | NO — replaced by `<DOSE_1>` |
| Grade 3 (AE grade) | NO — replaced by `<AE_GRADE_1>` |
| 15-Mar-2024 (date) | NO — replaced by `<DATE_1>` |
| 65-year-old (age) | NO — replaced by `<OTHER_UNIQUE_ID_1>` |
| female (sex) | **YES — remains in proxy text** |

**Assessment:** All six critical identifiers (site ID, compound code, dose, AE grade, full date, age band)
are correctly replaced with placeholders. Only the binary sex token "female" passes through.
Per HIPAA Safe Harbor, sex is not one of the 18 enumerated identifiers and is thus not caught by the
regex Safe Harbor layer; it is also not matched by any of the clinical detection patterns. This is a known
limitation — sex alone is a low-discrimination quasi-identifier — but it is a surface that could contribute
to re-identification in combination with other retained tokens. Rated PARTIAL per audit criteria.

## Overall verdict: PARTIAL

The backend passes four of five checks outright (3A, 3B, 3C, 3E) and scores PARTIAL on one (3D).
The critical privacy properties — no PHI in audit logs, canary detection before any outbound call,
MCP payload hashing, and ε monotonicity — all hold. The single PARTIAL finding is that the binary sex
token "female" is not stripped from proxy text, because sex is not enumerated in HIPAA Safe Harbor's
18-identifier list and is not covered by the clinical detection patterns. This is a minor quasi-identifier
leakage risk rather than a direct re-identification vector; no compound code, site ID, full date, or
age-year escapes the proxy layer. A future hardening pass should add sex/gender to the quasi-identifier
detection patterns. No FAIL conditions were triggered.
